# 04_EVAL_AND_GUARDRAILS.md
評価（CV）と提出事故防止の仕様

## 1. CV分割（リーク防止）
- 分割単位は **text_id / oare_id**（同一文書がtrain/valに混ざらない）
- train は `oare_id` のみなので、CVは `oare_id` 単位で固定
- 推奨：
  - CV（例：K=5）
  - 最終ホールドアウト（最後まで触らない）

## 2. ローカル評価（SacreBLEU互換）
- BLEU と chrF++ を同じ前処理で計算する（コーパス全体のマイクロ平均）
- `dp.eval` は「再現可能な実装」を第一にする
  - できれば SacreBLEU そのものを使用
  - 使えない場合は同等のトークナイズ・設定を合わせる

## 3. 反復の意思決定ルール
- LBを見てハイパラをいじるのは最終段だけ
- 施策採否は **CV平均 + ホールドアウト**で決める

## 4. 提出事故防止（validate）
`python -m dp.validate --submission submission.csv`
チェック項目（最低限）
- ヘッダが `id,translation`
- `id` が test の全件を含む（欠落/重複なし）
- `translation` が空でない
- `translation` に改行を含まない（\n, \r）
- `translation` が複数文になっていない（簡易：`. ` の過剰や `?` `!` の数で警告）
- CSVとして壊れていない（カンマ/クォートのエスケープ）

## 5. ガードレール（壊れやすい箇所のテスト）
`tests/test_preprocess_invariants.py`
- `<gap>/<big_gap>` が消えていない（前処理で“記号ごと消す”事故の検出）
- `{...}` / `(...)` determinatives が崩れていない（統一した場合も保持される）
- `[...]` / `…` / `xxx` 系の欠損が `<gap>/<big_gap>` に正規化されている
- ロググラム区切りの `.` が消えていない（例：`KÙ.BABBAR`）
- ALL CAPS を小文字化していない
- 変換マップ（プレースホルダ復元）が全件完結している

## 5.1 OCR追加データ用のガードレール
- candidates / pairs の各段階で統計を保存して監視する（件数、平均長、len_ratio分布、重複率）
- 次の兆候が出たら「追加データ混合を停止」し、候補抽出と品質ゲートを強化する：
  - CVが悪化し続ける
  - 固有名詞（大文字パターン）が崩れる
  - 出力が短文化/無内容化する
  - <gap>/<big_gap> の扱いが不自然になる
