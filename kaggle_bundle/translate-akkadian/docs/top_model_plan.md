Deep Past Challenge（アッカド語→英語）
トップ狙いモデル作成計画書

（対象：Old Assyrian / Akkadian transliteration → English）

# 1. 目的と成功指標

目的：古アッシリア語（アッカド語転写表記）を英語へ翻訳するニューラル機械翻訳（NMT）モデルを構築し、Leaderboard上位（トップ）を狙う。

評価指標：BLEU と chrF++ の幾何平均（コーパス全体でマイクロ平均）。

成功指標（運用上の定義）：

CV（text_id 分割）での指標改善が一貫して確認できること（LBより優先）

推論がKaggleのコードコンペ環境で安定・再現可能であること（時間/メモリ/乱数）

固有名詞・数値・単位・欠損表現（<gap>/<big_gap>）を破壊しないこと

# 2. ルール・制約（計画の前提）

本計画は、コンペページおよびルールに従って実行する。重要ポイント：

コードコンペ（提出はノートブック実行成果として採点）

外部データ／事前学習済みモデルは「自由かつ公開されているもの」に限り利用可能（規約の範囲内で活用）

提出ファイルは submission.csv 形式（id,translation）。translation は「1文」。

※最終的な解釈は必ずKaggle上の最新ルール文面を優先する（計画書は“実装方針”であり、規約の代替ではない）。

# 3. 全体戦略（勝ち筋）

このタスクは「モデル選定」よりも、データ整備と運用設計で差が出やすい。狙うべき勝ち筋は次の3本柱：

データの形をテストに寄せる：train（文書レベル）→文レベルへ再構成し、アライン品質を保証する

低リソース×表記ノイズに強い：byte/char 系を軸にし、固有名詞・欠損・記号を壊さない

反復回数を最大化：誤り分析→改善→再学習を高速で回す（LB追従ではなくCV主導）

# 4. 工程（フェーズ別・実施順）

## Phase 0：環境・再現性の土台作り

公式評価（BLEU/chrF++）の再現：SacreBLEU相当の実装で、ローカルCVと一致させる

実験管理：設定（YAML/JSON）、seed、学習ログ、モデル重み、前処理バージョンを一意に保存

提出パイプライン雛形：前処理→推論→後処理→submission.csv出力を最初に動かす

## Phase 1：データ監査（EDA）と前処理方針の確定

文字種・記号・大文字（固有名詞/ロググラム）・determinatives `{ } / ( )`・欠損表現（`[...]` / `…` / `xxx` / `<gap>`）の頻度監査

正規化は“最小限”から開始：Unicode統一（ダッシュ/アポストロフィ等）＋不要な現代注記を除去

前処理のA/Bテスト：正規化の強さ（弱/中/強）、ハイフン扱い、欠損マーカー保持方針を変えてCV比較

実データ差分の反映：train は改行/行番号がほぼ無い、published_texts は `<gap>/<big_gap>`、test に `„` が混入 → 正規化で表記を統一する

## Phase 2：粒度ギャップ解消（文書→文）と高品質アライン

train（文書レベル）を文レベルに分割：転写側は行番号前提にせず、target 文数に合わせた長さDPで分割

文アライン：長さ比＋文字n-gram類似（chrF的）＋lexiconヒントで対応付け

品質ゲート：極端な長さ比、文字化け、分割崩れ、重複を自動検出して除外

分割単位のCV：text_id単位でfold分割（リーク防止）＋最終ホールドアウト固定（意思決定用）

## Phase 3：強いベースライン（モデル2系統）

目的：後の改良が効く“壊れにくい”基準点を作る。

主力：byte/char 系 seq2seq（chrF++に強い）

副：subword 系 seq2seq（意味保持や流暢性で補完）

学習レシピ：label smoothing、長さ制約、早期終了、学習率スケジュール、gradient clipping を標準化

## Phase 4：固有名詞・数値・単位を落とさない設計

Lexiconで PN/GN 等を検出し、学習時はプレースホルダ化（例：<PN_1>）→生成後に復元

ALL CAPS（ロググラム）と先頭大文字（固有名詞）を保持し、正規化で潰さない

数値・単位・記号の後処理：英訳側の表記揺れを抑える（例：桁区切り、括弧、コロン等）

## Phase 5：追加並列データ（OCR出版物）を“安全に”増やす

publications.csv から翻訳候補を抽出：`has_akkadian` で一次フィルタ→識別子（aliases等）で探索→該当ページを絞る

英語以外は英語へ統一（ただし低品質は切り捨てる）

品質階層（高/中/低）を作り、混ぜ方を最適化：高のみ→高+中→（必要なら）自己学習

混合比率をCVで探索し、ノイズで壊れる兆候（固有名詞崩壊、脱文）を監視

## Phase 6：自己学習（翻訳なし8kを活かす）

published_texts（翻訳なし）をベースモデルで英訳→擬似並列データ化

信頼度フィルタ：長さ比、英語流暢性、固有名詞整合、重複除去、極端ケース排除

高信頼のみ追加して再学習（反復）

## Phase 7：文脈（document context）の導入

testの text_id で文書を再構成し、対象行の前後±k行を入力に付与（ターゲット行は特別トークンで囲う）

学習も同形式に合わせ、文脈から固有名詞・関係語・照応を補えるようにする

k（文脈幅）と最大長のトレードオフをCVで最適化

## Phase 8：デコード最適化とリランキング

beam/length penalty/禁止トークン/最大長などを最適化（1文制約を守る）

n-best生成→指標近似（chrF寄り）でリランキング（MBR的選択も検討）

チェックポイント平均（checkpoint averaging）で安定化

## Phase 9：アンサンブル（差を付ける最後の一押し）

異種アンサンブル：byte/char 系＋subword 系＋seed違い

候補集合から、固有名詞・数値の整合と指標近似で最終選択

計算制約に合わせて量子化/軽量化（必要な場合のみ。精度劣化に注意）

## Phase 10：提出の事故防止（自動バリデーション）

CSVの改行混入、引用符崩れ、空行、id欠落、複数文生成を検出するテストを用意

全件推論の時間/メモリを測定し、Kaggle環境で確実に完走する構成に固定

# 5. 検証設計（CV/ホールドアウト/意思決定）

分割単位：text_id で分割（同一文書の混入を防ぐ）

CVの役割：改善施策の採否はCVで判断（LBは参考程度）

最終ホールドアウト：最後まで“触らない”検証セットを固定し、過学習（LB合わせ）を防ぐ

アブレーション：正規化/文脈幅/デコード/混合比率の効果を1つずつ検証して残す

# 6. 主要リスクと対策

| リスク | 症状（起きること） | 対策（工程） |
| --- | --- | --- |
| 過度な正規化 | 固有名詞/ロググラム/限定符情報が消えて精度低下 | 正規化A/Bテスト、最小限正規化、固有名詞保持 |
| 追加データがノイズ化 | 学習が壊れる/出力が短文化 | 品質ゲート、階層化、混合比率探索 |
| リーク/分割不備 | CVは良いが本番で落ちる | text_id分割、最終ホールドアウト固定 |
| 文アライン不良 | 誤対応ペアが増える | 長さ比DP + 類似度、品質ゲート、サンプル目視 |
| デコード事故 | 複数文/改行/空行で提出が壊れる | 自動バリデーション、禁止トークン、長さ制約 |
| 計算資源不足 | 学習/推論が完走しない | 小さく強いモデル優先、軽量化は最後、計測で固定 |
